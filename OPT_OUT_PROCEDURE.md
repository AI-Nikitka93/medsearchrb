# OPT_OUT_PROCEDURE

## 1. Назначение документа
Этот документ описывает:
- как поддержка обрабатывает запрос врача `удалите меня из бота`;
- как обрабатываются запросы клиник;
- что обязан реализовать архитектор и разработка.

## 2. Важное различие
### 2.1. Врач
- Обычно врач — физическое лицо и субъект персональных данных.
- Для него применяется порядок реализации прав субъекта персональных данных по Закону РБ № 99-З.

### 2.2. КЛИНИКА
- КЛИНИКА как юридическое лицо не всегда является субъектом персональных данных.
- Поэтому запрос клиники на скрытие карточки, акции или выдержек из отзывов не всегда является DSAR-запросом.
- Но сервис может рассматривать такой запрос как деловое обращение и скрывать данные по внутренней политике риска.

### 2.3. Исключение
- Если карточка относится к ИП, частнопрактикующему врачу или содержит данные конкретного физического лица, применяется логика субъекта персональных данных.

## 3. Юридическая логика для публичных данных
- Мы обрабатываем ранее распространенные персональные данные врача, если они были опубликованы самим субъектом, с его согласия или в соответствии с требованиями законодательства.
- Однако при требовании субъекта о прекращении обработки и удалении и при отсутствии иных законных оснований обработка должна быть прекращена.
- Если у нас нет технической возможности полного удаления, мы обязаны заблокировать дальнейшую обработку.

## 4. Каналы приема запросов
- email: `[privacy@domain.by]`
- support-бот: `/optout`
- web-форма: `[URL]`

## 5. Что просим у заявителя
### 5.1. Для врача
- ФИО;
- ссылка на карточку в нашем сервисе;
- ссылка на исходный источник, если есть;
- суть требования: скрыть / удалить / исправить;
- данные для верификации личности.

### 5.2. Для клиники
- наименование клиники;
- должность и ФИО представителя;
- подтверждение полномочий представителя;
- ссылка на спорную карточку/акцию/выдержку;
- суть требования.

## 6. Минимально допустимая верификация
### 6.1. Врач
- письмо с официального email клиники, где врач работает;
- либо письмо/заявление с данными, достаточными для идентификации;
- либо электронный документ с подписью;
- либо иные доказательства, позволяющие разумно подтвердить личность.

### 6.2. КЛИНИКА
- письмо с корпоративного домена;
- доверенность / подтверждение полномочий;
- для ИП — документы, связывающие заявителя с карточкой.

## 7. Сроки
- первичное подтверждение получения запроса: до 2 рабочих дней;
- решение по существу: до 15 дней с момента получения надлежаще оформленного заявления;
- если запрос рискованный и явно затрагивает персональные данные врача, карточка может быть скрыта временно до завершения проверки.

## 8. Пошаговый процесс поддержки
### Step 1. Регистрация обращения
- создать тикет;
- присвоить тип: `doctor_opt_out`, `clinic_request`, `data_change`, `review_complaint`.

### Step 2. Временная защитная мера
- если запрос касается ФИО врача и выглядит правдоподобно, установить временное скрытие карточки:
  - `is_hidden = true`
  - `opt_out_status = pending`

### Step 3. Верификация
- проверить, кто заявитель;
- проверить, относится ли запрос к персональным данным;
- проверить, есть ли иные основания для дальнейшей обработки.

### Step 4. Юридическая квалификация
- `Врач / физлицо`:
  - если иных оснований нет, прекращаем обработку и скрываем карточку;
  - если есть иное законное основание, готовим мотивированный отказ со ссылкой на норму.
- `Клиника / юрлицо`:
  - если спор касается бренда, акции или текста, решение принимается по внутренней политике риска;
  - если спор касается персональных данных конкретного врача, применяем режим врача.

### Step 5. Техническое исполнение
- установить:
  - `is_hidden = true`
  - `opt_out = true`
  - `opt_out_requested_at`
  - `opt_out_reason`
  - `opt_out_verified_at`
  - `opt_out_source`
- удалить карточку из публичной выдачи;
- исключить карточку из бота, Mini App и канала;
- добавить suppression-запись, чтобы парсер не вернул карточку обратно;
- если данные уже попали в кэш, очистить кэш;
- если данные есть в бэкапах и немедленное удаление невозможно, зафиксировать блокировку повторной обработки и дождаться ротации бэкапов.

### Step 6. Ответ заявителю
- сообщить результат;
- если удаление выполнено частично, указать, что публичный показ прекращен, а остаточные следы в бэкапах будут устранены по циклу ротации;
- если отказано, указать конкретное правовое основание.

## 9. Политика по отзывам
### 9.1. Что можно показывать по умолчанию
- агрегированный рейтинг;
- количество отзывов;
- ссылку на источник;
- дату последнего обновления;
- короткий нейтральный фрагмент только если это не нарушает правила источника и не искажает смысл.

### 9.2. Что рискованно
- копировать полный текст отзывов;
- редактировать отзыв так, что меняется смысл;
- смешивать отзывы из разных источников без маркировки;
- публиковать отзывы без ссылки на источник;
- игнорировать условия использования конкретного сайта.

### 9.3. Безопасное правило для MVP
- По умолчанию показывать `рейтинг + число отзывов + источник`.
- Текстовые выдержки включать только после отдельной юридической проверки источников.

## 10. Обязательное требование к P-20 (Архитектору)
- В БД обязательно предусмотреть минимум:
  - `is_hidden BOOLEAN`
  - `opt_out BOOLEAN`
- Рекомендуемые поля:
  - `opt_out_status`
  - `opt_out_requested_at`
  - `opt_out_verified_at`
  - `opt_out_reason`
  - `source_url`
  - `suppression_key`

## 11. Обязательное требование к парсеру
- Перед публикацией парсер обязан проверять suppression-таблицу.
- Если сущность помечена `is_hidden = true` или `opt_out = true`, повторная публикация запрещена.

## 12. Шаблоны решений
### 12.1. Удовлетворение требования врача
`Карточка врача скрыта из публичной выдачи. Повторная публикация из парсера заблокирована.`

### 12.2. Частичное удовлетворение требования клиники
`Текстовые выдержки и акция скрыты. Карточка клиники и ссылка на официальный сайт оставлены как справочная информация.`

### 12.3. Безопасный отказ
`Запрос рассмотрен. Удаление не выполнено, поскольку обработка сохраняется на ином правовом основании. В ответе необходимо сослаться на конкретную норму законодательства.`
